# Kế hoạch triển khai Module 1 — Đặt lịch hẹn & Hàng chờ

## Context

Dự án `ClinicBooking` (`D:\Code\C#\DatLichPhongKham`) đã chốt split 4 module theo `docs/phan-chia-module-va-huong-dan-du-an.md`; Module 1 do **tôi + Claude** đồng sở hữu, 3 module còn lại do 3 thành viên khác phụ trách và chưa được đẩy lên GitHub. **Mục tiêu plan này**: lên kế hoạch để Module 1 tiến xa nhất có thể *trong lúc chờ team*, bằng cách:

1. Xác định những abstraction Module 1 cần từ Module 2/4 và **tự mình định nghĩa + implement stub** để handler chạy được.
2. Chia handler thành 3 lớp theo mức phụ thuộc stub (pure / non-critical stub / critical-path stub), để có thứ tự triển khai và priority test rõ ràng.

Sau 4 wave, Module 1 có thể ship endpoint chạy thật (với stub sau DI) trước khi team đẩy Module 2/4; chỉ cần swap DI registration khi code team lên là xong integration.

## Phân loại handler theo mức phụ thuộc stub

Tiêu chí phân wave (làm rõ theo phản hồi review):

- **Pure**: handler chỉ đọc/ghi table Module 1 + `IAppDbContext`, không gọi stub.
- **Non-critical stub**: handler có gọi stub nhưng là *side effect* (notification fire-and-forget; decrement counter best-effort). Nếu stub fail/không làm gì, flow chính vẫn đúng.
- **Critical-path stub**: flow chính phụ thuộc output thật của stub (availability check, atomic increment). Khi Module 2 chưa lên thì stub đang "đóng thế" — logic handler không đổi, chỉ cần chạy integration khi team lên.

## Quyết định thiết kế chính (fold vào advisor feedback)

1. **`LichHen.SoSlot` là ordinal position trong ca** — xác nhận từ `database/clinic.dbml` dòng 203. Unique index `(IdCaLamViec, SoSlot)` hợp lệ, là xương sống chống double-book.
2. **`GiuCho.SoSlot` chưa rõ ordinal hay quantity** — DBML mơ hồ. Không đặt unique index ở Wave 1; chốt với PM trước khi viết `TaoGiuCho` ở Wave 3.
3. **Slot concurrency**: atomic `ExecuteUpdateAsync` tăng `SoSlotDaDat` + unique index `(IdCaLamViec, SoSlot)` + retry tối đa 2 lần trên `DbUpdateException`. Không pessimistic lock.
4. **`SoThuTu` check-in**: `MAX+1` per `IdCaLamViec` + unique index + retry 3 lần. Tần suất thấp (thao tác thủ công lễ tân) → retry acceptable.
5. **"Huỷ muộn"** threshold từ `LichHenOptions.HuyMuonTruocGio` (default 24h). Handler so `(CaLamViec.NgayLamViec + GioBatDau) - UtcNow`.
6. **MaLichHen**: format `LH-{yyyyMMdd}-{seq6}`. Generator dùng `MAX(...)` parse + 1 (không COUNT — tránh drift khi có xoá). Unique index `MaLichHen` là backstop.
7. **Cross-module write `BenhNhan.SoLanHuyMuon`** — cột thuộc Module 3. Ghi qua `IAppDbContext.BenhNhan`. **Coordination point**: notify owner Module 3 để họ không đặt constraint/trigger conflict.
8. **GiuCho expiration**: Wave 1–3 dùng lazy cleanup (stub lọc `DaGiaiPhong==false && GioHetHan > UtcNow` + opportunistic `ExecuteUpdate` đánh dấu hết hạn). Wave 4 chuyển sang Hangfire job (Module 4).
9. **Gọi tiếp auto-complete luồng hiện tại**: option bool, default `true`.

## Testing strategy

- **Wave 1**: scaffold project `ClinicBooking.Application.UnitTests` (không viết test handler — chưa có handler).
- **Wave 2**: mỗi handler ≥ 1 test happy path + ≥ 1 test lỗi. `TestDbContextFactory` (SQLite `:memory:`) + NSubstitute cho `IDateTimeProvider`, `ICurrentUserService`, `ICaLamViecQueryService`, `INotificationService`, `IMaLichHenGenerator`. Dùng FluentValidation `TestHelper` cho validator.

Mỗi handler phải pass Definition of Done (doc tuần 6 đã liệt kê): command/query/handler/validator, DTO request/response, controller có `[Authorize]` + `[ProducesResponseType]`, không magic string, exception chuẩn, `dotnet build` không warning mới, `dotnet test` xanh, smoke test Swagger, commit Conventional Commits.

## Risks & open questions

- **Ngữ nghĩa `GiuCho.SoSlot`**: chờ PM xác nhận ordinal vs quantity — ảnh hưởng Wave 3 `TaoGiuCho`.
- **Module 2 có thể mô hình hoá `SoSlotDaDat` khác đi** (derived column, sự kiện domain) — `ICaLamViecQueryService.IncrementSoSlotDaDatAsync` là buffer, nhưng contract semantics cần review khi Module 2 lên.
- **Module 4 có thể yêu cầu `ThongBao` ghi cùng transaction với command gốc** — stub hiện làm fire-and-forget, review lại ở Wave 4.
- **Timezone**: giả định `CaLamViec.NgayLamViec + GioBatDau` là giờ địa phương (Asia/Ho_Chi_Minh). Cần helper `TimeZoneInfo.ConvertTimeToUtc` nhất quán — tạo ngay trong Wave 1 kèm `IDateTimeProvider` (mở rộng thành `DateTimeOffset`-aware).
- **Seed fixture data cho smoke test**: chưa có `CaLamViec` / `BacSi` / `BenhNhan` seed nào. Cần tạo 1 migration Module 1 riêng cho fixture dev *hoặc* script SQL + README, **không ship lên production**.
